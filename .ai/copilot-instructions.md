# Copilot 使用指南

本仓库包含一个基于 Three.js 的沙盒插件系统。

在生成或修改代码时，您必须遵守以下规则。

---

# 1. 任务执行规则

这些规则控制 Copilot 如何执行任务。

- 始终确保代码编译成功。

<!-- - 如改动沙箱相关代码
执行src\components\StarSky\Desktop\PluginPage\DevWorkbench\index.tsx autoRunPlugin方法确保加载插件无报错
或 需打开 http://localhost:3000/planet 并检查控制台无沙箱相关错误输出。 -->

- 绝不引入 TypeScript 错误。

- 如果已配置 ESLint，则绝不引入 ESLint 错误。

- 除非明确要求，否则保持现有架构不变。

- 仅修改所需的最少代码量。

- 除非注释错误，否则请勿删除现有注释。

- 请勿更改文件编码。

- 仔细执行每项任务，请勿遗漏。

- 如无法完成的任务请明确指出。

编码规则：

- 所有文件必须使用 UTF-8 编码。

- 注释必须支持中文字符，且不会出现乱码。

错误处理：

- 请勿忽略错误。

- 始终提供清晰的错误信息。

代码安全：

- 请勿使用 `eval`、`Function()` 或动态脚本执行。

---

# 2. 通用编码规则

语言规则：

- 尽可能使用 **TypeScript**。

- 使用强类型，避免使用 `any`。

- 类 类型 接口 成员定义 加上中文注释。重要的逻辑必须包含注释。

风格规则：

- 使用描述性的变量名。

- 不要出现重复的大段代码，尽量使用可重用的小方法。

- 方法尽量避免嵌套，保持结构清晰，提高可读性。

- 样式文件和组件文件分离，保持组件文件专注于逻辑和结构。

文档规则：

- 公共函数必须包含 JSDoc。

- 重要的逻辑必须包含注释。

示例：

```ts

/**
  * 创建一个立方体网格
  * @param size 大小
  * @param color 颜色
  */
function createCube(size: number, color: string): THREE.Mesh {}

```
api服务端接口规则：
每个api方法都需要明确强类型的输入参数及返回值，以及真实的页面数据传入。

# 3. 代码设计通用原则

在编写、修改或重构代码时，必须遵循以下设计原则。

## 1. 保持模块单一职责（Single Responsibility）
每个模块、类或函数应只负责一个明确功能。

错误示例：
一个类同时负责 UI、网络请求、数据存储。

正确示例：
UIController
DataService
NetworkClient
各自独立。

---

## 2. 保持代码结构清晰
避免复杂嵌套和难以理解的逻辑。

规则：
- 函数长度尽量小于 60 行
- 嵌套层级不超过 3 层
- 避免过度使用回调
- 优先使用 async/await

---

## 3. 优先使用组合而不是继承
尽量通过组合对象实现功能，而不是复杂继承结构。

错误示例：
DeepClassA → DeepClassB → DeepClassC

正确示例：
功能模块组合：
Renderer
Physics
Interaction
PluginManager

---

## 减少复杂度、精简内聚
保持代码单一简洁性，提高内聚协同，避免代码臃肿发散。

## 4. 明确的数据流
数据流应清晰可追踪，避免隐式状态。

规则：
- 尽量避免全局变量
- 状态应由明确模块管理
- 修改状态必须通过明确函数

## 7. 可测试性
代码应易于测试。

规则：
- 避免函数依赖隐藏状态
- 避免直接访问全局对象
- 使用依赖注入（Dependency Injection）

---

## 8. 错误处理必须明确
所有可能失败的操作必须处理错误。

规则：
- 网络请求必须 try/catch
- 插件执行必须隔离错误
- 不允许 silent failure

---

## 10. 不允许重复代码（DRY）
重复逻辑必须提取为函数或模块。

规则：
如果出现 2 次相似逻辑，应立即重构
